# Runbook: Operations & Incident Response

This document outlines standard operating procedures and emergency response strategies for maintaining the Forensic Council infrastructure.

## Service Commands

The infrastructure runs via Docker Compose in the root directory.

### Quick Commands

**Infrastructure Only (Redis, Postgres, Qdrant):**
*   **Start infrastructure:** `docker compose -f docker-compose.infra.yml up -d`
*   **Stop infrastructure:** `docker compose -f docker-compose.infra.yml down`
*   **View infrastructure logs:** `docker compose -f docker-compose.infra.yml logs -f`

**Full Stack (Development):**
*   **Start all services:** `docker compose up -d`
*   **Stop all services:** `docker compose down`
*   **Restart a specific service:** `docker compose restart <service_name>` (e.g., `backend`, `frontend`)
*   **View live logs of a service:** `docker compose logs -f <service_name>`
*   **Rebuild backend API:** `docker compose build backend`

**Production:**
*   **Start production stack:** `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`

## Incident Response Scenarios

### 1. Backend Memory Bloat (Redis Full)
If the backend starts rejecting requests with `OOM command not allowed` or begins severely lagging:
1.  **Diagnosis:** Check Redis memory usage: `docker exec -it forensic_redis redis-cli info memory`
2.  **Immediate Remediation:** Flush active investigator cache sessions.
    ```bash
    docker exec -it forensic_redis redis-cli FLUSHDB
    ```
    *Note: This will terminate any currently running live investigations and clear HITL checkpoints.*
3.  **Root Cause:** The `ex=86400` TTL limit may have failed on a bulk generation task. Report this as a bug.

### 2. WebSocket Hanging / Endless Processing
If an investigation stalls on "Processing Data Stream..." for more than 5 minutes:
1.  **Diagnosis:** An ML subprocess is likely hanging or zombie'd.
2.  **Mitigation:** Restart the API container to sever the hung Popen loop.
    ```bash
    docker restart forensic_api
    ```
3.  **Client action:** Users will see a "Connection lost" error and need to re-upload.

### 3. Signing Key Compromise
If the `SIGNING_KEY` is leaked or requires rotation:
1.  **Generate a new key:** `python -c "import secrets; print(secrets.token_hex(32))"`
2.  **Update Configuration:** Place the new key in the root `.env` file under `SIGNING_KEY`.
3.  **Restart the system:**
    ```bash
    docker compose up -d --force-recreate backend
    ```
4.  **Impact:** All *past* cryptographically signed reports generated by the old key will fail the new key's signature verification. This is intended behavior. The custody chain starts anew.

### 4. Database Corruption / Recovery
In the event that the PostgreSQL custody logs become corrupted:
1.  **Manual DB access:** `docker exec -it forensic_postgres psql -U forensic_user -d forensic_council`
2.  **Backup creation:**
    ```bash
    docker exec -t forensic_postgres pg_dumpall -c -U forensic_user > dump_$(date +%Y-%m-%d_%H_%M_%S).sql
    ```
3.  **Complete Reset (Destructive):**
    ```bash
    # For infrastructure-only setup
    docker compose -f docker-compose.infra.yml down -v
    docker compose -f docker-compose.infra.yml up -d
    
    # For full Docker setup
    docker compose down -v
    docker compose up -d
    ```
    *(The `-v` flag removes the `postgres_data` volume, wiping all evidence custody trails and Qdrant similarities.)*
